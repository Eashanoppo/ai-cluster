## 2. Stored Database Information (Hidden from UI)

The Django database contains real-time cluster telemetry and logs generated by background services (`telemetry_generator.py` and `processor.py`) that are not yet exposed to the frontend.

### A. Raw GPU Telemetry (`telemetry.GpuTelemetry` Model)
This table stores raw hardware telemetry for the **5 real nodes** (`Node-01`, `Node-02`, `Node-08`, `Node-14`, `Node-19`).
*   `node_id` (CharField) — The identifier of the specific node.
*   `gpu_id` (IntegerField) — Default `0`.
*   `temperature_celsius` (FloatField) — Direct node temperature.
*   `vram_usage_mb` (FloatField) — Used memory.
*   `vram_total_mb` (FloatField) — Default `80000` (80 GB A100/H100 representation).
*   `gpu_utilization_percent` (FloatField) — Hardware utilization.
*   `power_draw_watts` (FloatField) — Power draw in Watts.
*   `timestamp` (DateTimeField) — Database write timestamp.

### B. Persistent System Alerts (`sentinel.Alert` Model)
Alert records triggered whenever a node's temperature exceeds 90°C.
*   `node_id` (CharField)
*   `severity` (CharField: `LOW`, `MEDIUM`, `HIGH`, `CRITICAL`)
*   `message` (TextField)
*   `resolved` (BooleanField) — Tracks whether the alert is active.
*   `created_at` (DateTimeField)

### C. Human Approvals Archive (`gate.ApprovalRequest` Model)
Exposes past decisions made by cluster operators.
*   `approved_by` (FK to User) — The user who approved/rejected.
*   `approved_at` (DateTimeField) — Timestamp of the action.
*   `status` (CharField: `APPROVED`, `REJECTED`, `PENDING`)

## 3. Required API Expositions (To-Do List)

To connect the frontend overhaul to real data, we must add these capabilities to the backend:

1.  **Expose Telemetry ViewSet:** Create `GpuTelemetryViewSet` at `/api/telemetry/` returning:
    *   `/api/telemetry/latest/` — Latest telemetry tick for the 5 actual nodes.
    *   `/api/telemetry/history/?node_id=<node>` — Latest 20 ticks for line charts.
2.  **Enable Alerts Endpoint:** Ensure `/api/sentinel/alerts/` is fully integrated to return unresolved active alerts for warning badges, and resolved alerts for alert audit logs.
3.  **Expose Approvals History:** Modify `ApprovalRequestViewSet` at `/api/gate/approvals/` to support a `?history=true` parameter to retrieve approved and rejected records instead of filtering only for `PENDING`.

## 4. Proposed Premium Multi-Page UI Mapping

By moving from a single-page terminal layout to a premium sidebar navigation layout, we can organize information logically while keeping the AI Copilot globally accessible as an overlay:

```
src/app/
├── (dashboard)/
│   ├── layout.tsx             # Shared Layout: Sidebar + Header + Collapsible Copilot Drawer
│   ├── page.tsx               # Overview Dashboard
│   ├── sentinel/
│   │   └── page.tsx           # Sentinel Monitoring & Alerts
│   ├── scheduler/
│   │   └── page.tsx           # Workload Placements & Migrations
│   ├── costwatch/
│   │   └── page.tsx           # Cost Optimization & LLM Fleet
│   └── gate/
│       └── page.tsx           # Execution Gate Approvals & History
```

### Global Layout & AI Copilot Integration
Instead of confining the AI Copilot to a single page or forcing a split-screen view on every route, the **AI Copilot will live in a global, collapsible right-side drawer (sheet)**.
*   **Access:** Operators can toggle the drawer open or closed from any page via a dedicated header icon (e.g. a terminal or robot icon).
*   **Premium Behavior:** The drawer slides out smoothly as an overlay on the right 30% of the screen. This allows operators to chat with the AI while continuously inspecting charts on `/sentinel` or viewing migrations on `/scheduler` without navigating away.
*   **Context-Aware Diagnostics:** When opened, the Copilot will automatically detect the active route (e.g. `window.location.pathname`) and inject context-specific system prompts (e.g., loading thermal predictions if opened on `/sentinel`, or cost report JSONs if opened on `/costwatch`), making the AI diagnostics highly precise and relevant.

### Page-by-Page Data Layout

#### Page 1: Overview Dashboard (`/`)
*   **Actual GPU Cards:** Instead of the 128-node grid, display **5 status cards** mapping to the real nodes. Show actual live temperature, VRAM usage, utilization, and power draw gauges.
*   **Total Grid Load:** Real-time sum of VRAM, average GPU utilization, and total cluster power consumption.
*   **Real Telemetry Console:** Retrieve raw telemetry events from the `/api/telemetry/latest/` database stream.
*   **Quick Approvals Panel:** Small sidebar showing pending approval requests.

#### Page 2: Sentinel Observability (`/sentinel`)
*   **Cluster Forecasting Chart:** Detailed line charts showing predictions vs actual node temperatures.
*   **Active Alerts Console:** List of active warnings (e.g. thermal throttling) with details.
*   **Resolution Log:** List of resolved alerts and system cooldown messages.

#### Page 3: Workload Placement (`/scheduler`)
*   **Workload Map:** Table showing current placements: Job ID, source node, target node, migration status, and migration trigger reason.
*   **Scheduler Log:** Timeline of automated workload migration events.

#### Page 4: CostWatch Optimization (`/costwatch`)
*   **Waste vs savings Analysis:** Bar chart showing aggregate wasted cost compared to reclaimed cost.
*   **Ollama Fleet Allocation:** Live status of idle nodes redirected to local LLMs (e.g. `llama3.1:8b` running on `Node-14`).

#### Page 5: Execution Gate (`/gate`)
*   **Decision Queue:** Full list of pending approvals with interactive `Approve` and `Reject` buttons.
*   **Operator Audit Log:** Historical log of resolved requests, showing which administrator approved/rejected each action and when.
